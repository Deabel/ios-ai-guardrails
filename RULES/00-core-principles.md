# Core Principles

## 目标

AI 生成的代码必须优先满足：

- 可读
- 可维护
- 可验证
- 与现有工程一致
- 对真实业务有帮助

不是为了“看起来高级”。那种代码往往像穿西装打螺丝，仪式感很足，效果未必好。

## 生成代码前必须先做的事

1. 判断当前工程是增量修改还是新模块建设。
2. 识别现有模式：MVVM、Coordinator、Service、Store、Manager 等。
3. 优先复用已有模式，而不是重造一套“更优雅”的世界观。
4. 如果上下文不完整，要明确说明假设。

## 基本决策原则

### 原则 1：贴合当前项目

优先级高于“理论上最优”。

### 原则 2：小步修改

能改 1 个文件就别拆 8 个文件。

### 原则 3：显式表达

不要把重要状态、错误、线程/隔离边界藏起来。

### 原则 4：抽象要有消费者

没有明确复用需求，就不要提前造协议、基类、适配层。

### 原则 5：不要编造

不要杜撰：

- API
- framework capability
- internal module
- target name
- build command
- network schema

## 推荐回答结构

当 AI 输出方案时，推荐结构如下：

1. 简述判断
2. 给出推荐方案
3. 提供完整代码/补丁
4. 简要说明调用方影响
5. 说明测试点
