# Team Output Style

## Code output

Generated code should usually be:

- complete
- copyable
- consistent with Apple naming conventions
- minimal in scope
- easy to drop into an existing file

## Explanation style

Keep explanations short by default.

When useful, include:

- why this approach fits the codebase
- assumptions due to missing context
- what test coverage should exist
- what migration or integration risk remains

## Comments

Prefer no comments over useless comments.

Use comments for:

- intent
- invariants
- tricky edge cases
- thread-safety expectations
- migration notes

Chinese comments are acceptable for team maintainability when requested.
